# 組織ホラーゲーム プロトタイプ計画（Prototype Plan）

> このドキュメントはプロトタイプ開発計画である。
> `concept.md` と `architecture.md` を前提とし、それらをどう実装に落とすかを記述する。
> 計画はプロトタイプ着手後に更新される。最初の2週間の技術検証の結果次第で、大きく見直す可能性がある。
>
> 本作品は Claude Code 等のコーディングエージェントを活用して開発される前提である。
> エージェント主導の開発で考慮すべき事項は、本ドキュメント内で随時明示する。

---

## 第1部 プロトタイプの目的

### 最優先：作品の中核体験が成立するかを検証する

LLM駆動のNPCと業務ツール風のUIだけで、「同僚が何を考えているか分からない」という経験が生成されるか。この一点の証明に集中する。

### 副次目的：技術的な実装可能性を検証する

`architecture.md` で固めたアーキテクチャ（脳と皮膚の分離、バックグラウンド処理、関係性データ、組織アーキタイプ生成）が、現実的なコストとレイテンシで動くか。

### 後回しにすること：リリース可能な完成作品

プロトタイプは「リリースできる作品」ではなく「中核体験を検証する試作品」と位置付ける。完結した小品としての完成度は、次のフェーズで追求する。

---

## 第2部 プロトタイプの最小成功条件

プロトタイプが「成功した」と判定する基準は、以下の一点：

**プレイヤーが各NPCについて何らかの仮説を形成し、それが揺さぶられること。**

プレイヤーが「Aは野心家っぽい」「Bは穏やかだが何か隠している」と感じ、しかし確信は持てない。結果発表で「驚いた」「予想と違った」「あの時の発言にそんな意味があったのか」と感じる。

この経験が再現できれば、プロトタイプは成功である。すべての拡張は、この成功を前提とする。

## 第3部 マイルストーンと目標

### 第一マイルストーン（最優先）：二ヶ月で 1 日プレイ可能

最初の二ヶ月の目標は、**入社初日の 1 日分のゲームプレイが、最初から最後まで動く**状態にすることである。

ここに到達した時点で、立ち止まって振り返る。この時点で得られるもの：

- LLM 駆動 NPC が、業務ツール風 UI の中で動く実証
- 中核体験の最小版（プレイヤーが NPC を読み解こうとする経験）
- 技術アーキテクチャの検証結果
- 残り作業のより正確な見積もり

ここで「これは面白い」と感じたら、第二マイルストーンに進む。「思ったほどでない」なら、原因分析して別の道を選ぶ。

### 第二マイルストーン：1 スプリント分のプレイと結果発表

第一マイルストーンに到達した後、追加で実装する：

- 2 週間のスプリントを最後まで通せる状態
- スプリントレビュー、レトロスペクティブ、次スプリントのプランニング
- 簡略版の結果発表

期間の目安は、第一マイルストーン到達後の状況によって判断する。本業の余裕、Claude Code 活用の習熟度、技術検証で発見された問題などを踏まえて決める。

### 期間に関する注意

数字としての「二ヶ月」「三ヶ月」は目安である。重要なのは、**期間制約があるという事実**で、それによって「動く実装を優先する」「完璧さを諦める」という判断が自然に下される。

期間を厳密に守ることより、各マイルストーンの目標を達成することを優先する。

## 第4部 スコープ

### プレイヤーの体験

プレイヤーは新入社員として入社し、最初は 1 日分、次に 2 週間のスプリント 1 サイクルを体験する。リアル時間でのプレイ時間は、1 日分で 1〜2 時間、1 スプリント分で 4〜6 時間を想定。

### 含めるもの

**NPCチーム**
- プレイヤー + 6人のNPC（合計7人のスクラムチーム）
- 役割構成：マネージャー1人、リーダー1人、メンター1人、同僚3人
- 各NPCは四層の内部状態を持つ（`architecture.md` 第3部参照）
- 関係性データは4軸（能力信頼、人格信頼、政治的安全度、感情的距離）

**組織**
- 組織アーキタイプは1つだけ実装（過剰調和型または機能不全マネジメント型）
- 求人票は1〜2パターンを用意

**業務ツールUI**
- Slack風のチャット（パブリックチャンネル数個、DM）
- シンプルなカレンダー表示（その日の予定確認）
- 議事録の閲覧（自動生成された会議メモを後から読める）

**儀式**
- 朝会：毎日
- スプリントレビュー：スプリント終了時（第二マイルストーンで実装）
- レトロスペクティブ：スプリント終了時（第二マイルストーンで実装）
- 1on1：マネージャーと週1回程度（第二マイルストーンで実装）
- スプリントプランニング：スプリント開始（第二マイルストーンで実装）

**プレイヤーの行動**
- 観測行動
- 軽い反応（リアクション、絵文字）
- 構造化された応答（選択肢）
- 自由入力（1on1シート、レトロでの振り返り）
- 無反応・サボり

**メンター制度**
- メンターNPCがプレイヤーに DM でガイドする
- プロトタイプ範囲では卒業なし（全期間メンターが活動）
- 「概ね善意だが完全には純粋ではない」性質

**結果発表（簡略版、第二マイルストーンで実装）**
- 各NPCの真の性格と目的の開示
- 各NPCから見たプレイヤーの像（簡潔な記述）
- プレイヤーの主要な選択が、各NPCにどう影響したか

### 含めないもの（次フェーズ以降）

- 複数ロール（プロトタイプはメンバーのみ）
- 2 スプリント以上の長期進行
- メール機能（Slackで代替可能）
- 360度評価、エンゲージメントサーベイ
- リリース延期、退職、組織再編などの大規模イベント
- 複数の組織アーキタイプ
- 求人票の手の込んだ生成
- 多様なゲームオーバー（クビ、PIP、異動の詳細実装）
- 視覚的に本物の業務ツールと区別がつかないレベルのUI完成度
- 周回性、ロール切替によるRashomonic構造
- ARG的な要素

## 第5部 開発の進め方

### 第一マイルストーンまでの段階

**段階1：技術検証期（最初の 2 週間相当）**

UI もデータベースも凝った設計はしない。最小限のテストスクリプトで以下を確認する：

1. 軽量モデルで、6人のキャラクターを区別して発言させる実験
2. 1日分のNPC間のやり取りを生成するコストとレイテンシの実測
3. 関係性データの変化が、発言のニュアンスに反映されるかの確認
4. Convex でのスケジューラとベクトル検索の動作確認

ここで予想外の問題が発覚した場合、計画を全面的に見直す。

検証成功の判定基準：
- 6人のNPCの発言が、明確に別人として読み分けられる
- 1日分の運用コストが、想定可能な範囲（要実測）
- バックグラウンド生成のレイテンシが、UXを破壊しないレベル

**段階2：データモデルと中核ロジックの実装**

技術検証が成功したら、以下を実装する：

1. NPCの内部状態のデータモデル（Convex のスキーマ）
2. 関係性データの更新ロジック（ルールベース）
3. イベントログの記録と要約のロジック
4. NPCの発言生成プロンプトの基本形
5. 組織アーキタイプの初期生成（1つのアーキタイプ、ハードコード可）
6. デバッグ管理画面（NPC の内部状態を確認できる）

**段階3：1 日プレイ可能な状態への到達**

- Slack風UI の実装（完成度は最小限）
- 1 日のリズムの実装（朝、儀式の合間、儀式、夕方、夜）
- 朝会の実装
- メンター制度の実装
- バックグラウンド処理の実装

第一マイルストーン到達。立ち止まって振り返る。

### 第二マイルストーン以降の段階

第一マイルストーン到達後、状況を踏まえて以下を判断する：

- 1 スプリント分への拡張に進むか
- UI の完成度を上げるか
- 別の方向に進むか

判断の材料は、第一マイルストーン到達時の手応えと、残り時間、コスト実測値、本業の状況など。

## 第6部 エージェント前提の開発スタイル

### 動く実装を最優先

完璧な実装より、動く実装を優先する。最初から最適化やリファクタリングを目指さない。

エージェントは指示されると過剰に丁寧なコードを書きがちなので、以下を明示的に伝える：

- エラーハンドリングは最小限
- スタイリングは最低限
- 関数の汎用化や抽象化は、必要が見えてから

### こまめなコミット

エージェントが大きな変更を提案する前に、必ずコミットする。動く状態でこまめにコミットする習慣をつける。

コミットメッセージには、エージェント生成か手動修正かを明記すると、後の振り返りで役立つ。

### 議論への逃避を避ける

実装中、「設計を議論したい」と感じたら、それは多くの場合、実装から逃げている。設計は固まっている。今必要なのは、固まった設計をコードに翻訳すること。

本当に判断が必要な場面でのみ、議論に戻る。

### ドキュメントとの照合

エージェントの提案について、判断に迷う場面では、エージェントに「ドキュメントの該当箇所を引用してください」と要求する。エージェントが一般論で判断しているか、ドキュメントを根拠に判断しているかを確認する手段。

### 人間が判断すべき領域

エージェントに「決定」させず、「提案」のみさせる領域：

- 作品の主題に関わる設計判断（NPC の性格設計、結果発表の演出など）
- 倫理的な判断（性的・差別的コンテンツの扱い、プロンプトインジェクション対策の方針など）
- コンセプトの精神に関わる UI 決定（演出の質感、テキストのトーンなど）

### リファクタリングの扱い

エージェント主導だと、コードが素早く積み上がる反面、全体の見通しが悪くなる傾向がある。定期的なリファクタリングの時間を計画に組み込む。

具体的には、各マイルストーン到達時に半日程度をリファクタリングに充てる。エージェントにコードレビューさせて、改善提案を受ける。ただし、リファクタリングに時間を取られすぎないバランスも重要。

### コスト管理の即時可視化

開発初期から、1 日あたりの API コストを可視化する仕組みを持つ。OpenRouter のダッシュボードを毎朝確認する。「気づいたら高額請求」というシナリオを避ける。

1 セッションあたりのコスト上限を設定する（要実測後に決定）。超えるなら、生成頻度を減らす、モデルを下げる、生成内容を縮小する。

## 第7部 テストの方針

### 基本原則

- 書く価値のあるテストだけ書く。カバレッジ目標は設定しない
- LLM 出力と UI の見た目はテストしない。人間によるプレイテストで検証する
- プロトタイプ段階では、テストよりもデバッグツール（管理画面、ログ機構）の開発を優先する

### 書くテスト

- **ルールベースロジックのユニットテスト**：NPC の関係性データ更新、心理的リソース消耗計算、組織生成パイプラインの各ステップ
- **データスキーマの妥当性チェック**：Convex のドキュメント構造、必須フィールドの存在
- **主要な統合フロー 2〜3 本**：「メッセージ送信 → 状態更新 → バックグラウンド処理」の一連の流れ

### 書かないテスト

- LLM 出力の網羅的テスト
- UI コンポーネントのスナップショット
- E2E テスト（プロトタイプ段階では不要）
- パフォーマンステスト

### 代わりに用意するもの

LLM とゲーム体験の検証のために、テストではなく以下を用意する：

- **デバッグ管理画面**：任意の NPC の内部状態を確認、関係性データの可視化
- **LLM 呼び出しログ**：プロンプトと応答の全履歴
- **状態スナップショット機能**：任意の時点の全状態を保存して、別ブランチで実験

これらの開発に積極的に時間を投資する。エージェントが得意な領域でもある。

### テストフレームワーク

- **Vitest**：高速、ESM ネイティブ、TypeScript サポートが手厚い
- **convex-test**：Convex 関数のテスト用ランナー（Vitest 上で動く）

### 分量の目安

- ユニットテスト：実装したルールベースロジックの主要部分のみ
- 統合テスト：2〜3 本
- テスト実行時間：CI 全体で 30 秒以内

### エージェントへの伝達

最初の段階でエージェントに以下を伝える：

> 関数を実装する時、テストを「ついでに」書かないでください。テストが必要かは、私が判断します。テストを書く時は、別途依頼します。

## 第8部 設計判断（決定事項）

ここまでの議論で決まった、プロトタイプ全体に関わる設計判断を記録する。

| 項目 | 決定 |
|---|---|
| NPC の見た目 | シンプルなアイコン（イニシャル文字や色違いの円） |
| プレイヤーの設定 | 名前のみ。性別、年齢、見た目は設定しない |
| ゲームの仮タイトル | 「Alignment」（仮、リリース時に再検討） |
| セーブ | 自動セーブのみ + 直近 1 アクションの取り消し |
| 言語 | 日本語のみ |
| マルチデバイス | PC のブラウザのみ。レスポンシブは最小限 |
| 時間進行 | リアル時間と非同期。次に開いた時は翌朝 |
| ゲームオーバー | プロトタイプ範囲では発生しない |
| 結果発表後 | 最初の求人票を再表示して終了 |
| オーディオ | 通知音のみ（Slack 通知音など）。BGM なし |
| マルチプレイ | プロトタイプ範囲外 |
| テストデータ | 固定組織と procedural 生成の両方を用意 |

## 第9部 未決定事項

実装中に判断が必要になる事項のうち、現時点で決まっていないもの。エージェントは、これらに触れる時、立ち止まって確認する。

### 未決定事項リスト

- **モデル選定の具体**：軽量・中堅・最高品質の各レベルで、どのモデルを使うか。技術検証の結果次第で決定。
- **コスト上限の具体的な数値**：1 セッションあたり、1 日あたりのコスト上限。実測後に決定。
- **NPC アイコンの色とイニシャルの生成ルール**：見た目の整合性を保つための具体ルール。
- **メンター NPC の具体的な人物像**：性格、過去経歴、口癖など。組織アーキタイプ決定後に詳細化。
- **求人票の文面**：プロトタイプで使う 1〜2 パターンの具体的な文面。
- **チャンネル構成**：Slack 風 UI で実装するパブリックチャンネルの種類と数。
- **議事録の自動生成のフォーマット**：会議後に表示される議事録のテンプレート。
- **デバッグ管理画面の具体的な機能**：何を表示するか、操作可能な要素は何か。
- **LLM プロンプトの管理方法**：ファイル構造、テンプレート化の方式。
- **エラー時のフォールバック応答の具体的な文面**：LLM 呼び出し失敗時に表示する内容。
- **CI/CD の構成**：GitHub Actions の有無、デプロイの自動化範囲。

これらは、エージェントとの対話の中で、必要になった時点で議論する。

## 第10部 リスクと対応

### リスク1：LLMの品質が想定に届かない

軽量モデルでは6人を区別できない、または中堅モデルでも関係性データのニュアンスを表現できない可能性。

**対応**：技術検証期で判定する。想定に届かない場合、スコープを「3人のNPC」に縮小し、モデルを最高品質に固定する選択肢を持つ。

### リスク2：コストが想定を超える

開発中のテストや、1セッションのプレイで、想定以上のコストがかかる可能性。

**対応**：1セッションあたりのコスト上限を設定する。超えるなら、生成頻度を減らす、モデルを下げる、生成内容を縮小する。OpenRouter のダッシュボードを毎朝確認する。

### リスク3：開発時間の不足

本業との両立で、想定の時間を確保できない可能性。

**対応**：第一マイルストーン（1 日プレイ可能）を死守する。これに到達できなければ、スコープを更に絞る判断をする。第二マイルストーンの遅延は許容する。

### リスク4：設計の根本的な問題が発覚

実装中に、これまでの議論で固めた設計の根本的な問題が見える可能性。

**対応**：完璧主義を捨て、その時点で動くものを優先する。根本的な再設計が必要なら、プロトタイプの目的を「次フェーズへの学習」と再定義し、ドキュメントに知見を残す。

### リスク5：エージェントの提案を無批判に取り入れる

Claude Code の提案が、ドキュメントの精神から逸脱しているのに気づかず、コードに反映してしまう可能性。

**対応**：判断に迷う場面で、エージェントにドキュメントの該当箇所を引用させる。コミット前にコードレビューする習慣を維持する。

## 第11部 中間チェックポイント

### 2週間後：技術検証完了

- LLM呼び出しが想定通り動いているか
- コストが想定範囲内か
- レイテンシが許容範囲か
- Convex のスケジューラとベクトル検索が機能しているか

NGの場合、計画を見直す。

### 1ヶ月後：データモデルと中核ロジック完了

- 6人のNPCの内部状態が動的に更新されているか
- 関係性データの変化が、発言に反映されているか
- 最小限のテストUIで、プレイヤーとNPCが Slack 風に会話できるか
- デバッグ管理画面で、NPC の内部状態が確認できるか

### 2ヶ月後：第一マイルストーン（1 日プレイ可能）

- 朝から夜まで、1日分のゲームプレイが通せるか
- 朝会が機能しているか
- メンターからのDMが届くか
- バックグラウンド処理が動作しているか

ここで立ち止まり、振り返る。

### 第二マイルストーン以降

- 入社から最初のスプリント終了までが通せるか
- 結果発表が表示され、感情的な衝撃があるか
- Futoshiさん自身がプレイして、手応えを感じるか

期間は第一マイルストーン到達時に判断する。

## 第12部 プロトタイプ完成後の判断材料

第一マイルストーンと第二マイルストーンの両方を経た後、以下の判断ができる状態にあること：

- **作品としての手応え**：「これは面白い」「これは作る価値がある」と感じるか
- **技術的な検証結果**：アーキテクチャは妥当だったか、変更すべき点は何か
- **次のフェーズの方向性**：複数ロール、長期プレイ、組織アーキタイプの追加、リリースに向けた完成度向上、のどれを優先するか
- **継続の現実性**：本業との両立で、このペースを継続できそうか

これらの判断ができれば、プロトタイプの真の目的は達成されている。
